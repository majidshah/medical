# MedVault — Architecture & Conventions Spec

Companion to `CLAUDE.md`. This is the reference document for how MedVault is structured
and why. `CLAUDE.md` is the short operational brief; this is the detail behind it.

---

## 1. System overview (MVP)

Three independent tiers, communicating over HTTPS REST:

```
Browser (React SPA)  ——HTTPS——>  FastAPI REST API  ——async SQL——>  PostgreSQL
        |                              |                               |
   static host (CDN)            VM or PaaS host                managed Postgres
```

The React app is a static bundle served from a CDN. It is a pure client of the REST API.
The future React Native app will be a second, equally independent client of the same API.
This independence is the core architectural commitment — honour it in every decision.

## 2. FHIR resource mapping

Every clinical entity maps to a FHIR R4 resource. Store normalized in Postgres; expose
and export as FHIR. Keep all mapping in the backend `/fhir` layer.

| Domain entity            | FHIR R4 resource         | Coding system        |
|--------------------------|--------------------------|----------------------|
| Patient profile          | Patient                  | —                    |
| Medical condition        | Condition                | SNOMED CT            |
| Medical event / injury   | Procedure                | SNOMED CT            |
| Past treatment           | Procedure (treatment)    | SNOMED CT            |
| Medication               | MedicationStatement      | (RxNorm/local later) |
| Allergy                  | AllergyIntolerance       | SNOMED CT            |
| Immunization             | Immunization             | Pakistan EPI         |
| Family disease history   | FamilyMemberHistory      | SNOMED CT            |
| Lifestyle habit          | Observation              | LOINC                |
| Lab/imaging test result  | Observation / DiagnosticReport | LOINC          |
| Free-form preferences    | patient_detail JSON      | —                    |

The `fhir.resources` Python library (Pydantic-based) is the recommended way to construct
and validate FHIR resources on the way out and into export bundles.

## 3. Medical ID rules

- If a patient has a CNIC, the Medical ID **is** the CNIC, normalized to the canonical
  form `XXXXX-XXXXXXX-X` (e.g. `42201-1234567-8`).
- Dependents without a CNIC get an auto-generated ID: `{guardian_CNIC}-D{n}`, where `n`
  is a per-guardian sequential integer (D1, D2, ...). Example: `42201-1234567-8-D1`.
- Medical IDs are unique system-wide — enforce with a DB unique constraint, not just app
  logic.
- Medical IDs are searchable (indexed).

---

## 4. Data scoping & multi-tenancy

- Top-level tenant is the **account** (one login).
- An account owns many **patient profiles** (self + family members).
- Every patient-data query is scoped to the authenticated account. A query that can
  return another account's data is a security bug, full stop.
- The audit log records (actor account, action, target patient, timestamp) for every
  read/write of patient data.

---

## 5. API conventions

- Base path `/api/v1`. Breaking changes go to `/api/v2`, never mutate v1 contracts.
- Resource-oriented routes, plural nouns: `/api/v1/patients`,
  `/api/v1/patients/{id}/conditions`.
- Standard HTTP verbs and status codes. Validation errors → 422 with Pydantic detail.
  Auth failures → 401; scope/permission failures → 403; missing → 404.
- Pagination on all list endpoints (limit/offset or cursor) — never return unbounded
  lists.
- OpenAPI is auto-generated by FastAPI at `/docs` and `/openapi.json`. The frontend's
  typed client is derived from this — the contract is the source of truth.

---

## 6. Auth model

- Registration: email + password (argon2/bcrypt hashed).
- Login returns a short-lived access JWT and a longer-lived refresh JWT.
- Access token carries the account identity; every protected route resolves it to the
  account and scopes all data access accordingly.
- Refresh endpoint issues new access tokens; refresh tokens are revocable.
- Secrets (JWT signing key) from env vars only.

---

## 7. Lab/imaging report module detail

- Upload: store the original image/PDF in object storage; persist a reference + metadata
  in Postgres. No OCR in MVP — the file is a visual reference only.
- Manual entry: user picks a test from the pre-loaded Pakistani LOINC-mapped catalogue
  (each catalogue entry carries local reference ranges), then enters value, unit, date,
  lab name. Multiple test results can link to one uploaded report.
- Detail view: uploaded file shown side-by-side with structured results.
- Timeline: chronological list, filterable by date and category (lab/imaging).
- Trends: per-test line chart over time with reference-range overlay.
- Normality: colour-code results (in-range/out-of-range) against catalogue reference
  ranges.

---

## 8. CI/CD & branching

- Platform: GitHub Actions.
- Branching: trunk-based. `main` protected (require PR + passing CI, no direct pushes).
  Short-lived `feature/<name>` branches.
- Pipeline stages: install → lint → test → build (backend image + frontend bundle).
  Deploy stage added at the deploy phase; production deploy is a manual-approval gate.
- Every PR runs the full lint+test suite. No merge on red.

---

## 9. Environments

- `local` (Codespace): runs against a dev Postgres (managed free tier or container).
- `production`: managed Postgres, API on VM/PaaS, frontend on CDN. All config via env
  vars per environment. No environment-specific code branches — only config differs.

---

## 10. Definition of done (per slice)

A slice is done when: schema migration exists; service logic implemented; API endpoint
exposed and documented in OpenAPI; frontend screen wired to it; tests written and green;
linters clean; access scoping verified; and a short human-review note written for
anything touching auth, deletion, or data scope.
